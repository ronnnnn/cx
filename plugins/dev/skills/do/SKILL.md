# 並列タスク実行ワークフロー

引数で渡された複数タスクを並列で実行し、結果を集約して報告する。

## 重要な原則

1. **タスク内容に応じてアプローチを自動選択** - 単一セッション内の並列 agent または 複数 agent
2. **各タスクは独立して実行** - タスク間の依存関係がある場合は順序を考慮
3. **結果を集約して報告** - 全タスクの完了後にサマリを提示
4. **失敗したタスクは明示的に報告** - 成功・失敗を区別して報告

## 作業開始前の準備

**必須:** 作業開始時に `update_plan` で実行ステップを整理し、以下のステップを登録する:

```
update_plan({
  plan: [
    { step: "タスクの分析と分割", status: "pending" },
    { step: "アプローチ判定", status: "pending" },
    { step: "タスクの並列実行", status: "pending" },
    { step: "結果の集約と報告", status: "pending" },
  ]
})
```

各ステップの開始時に `update_plan` で `in_progress` にし、完了時に `completed` に更新する。

## 実行手順

### 1. タスクの分析と分割

引数で渡されたテキストからタスクを抽出する:

- 番号付きリスト (1. 2. 3.)、箇条書き (- / \*)、「と」「、」区切り等のパターンを認識
- 各タスクを独立した作業単位に分割
- タスク間の依存関係を検出 (「A の結果を使って B」等)

**分割結果を提示:**

```markdown
## 検出されたタスク

1. **タスク名** - 概要
2. **タスク名** - 概要
3. **タスク名** - 概要

依存関係: なし / タスク 2 はタスク 1 の完了後に実行
```

### 2. アプローチ判定

#### 2-1. 複数 agent の利用判断

- ユーザーが明示的に delegation / sub-agents / parallel agents を求めている場合のみ、**パターン B (複数 agent)** を検討する
- それ以外は **パターン A (並列 agent)** を基本とする

#### 2-2. アプローチの選択

複数 agent が利用可能な場合、以下の基準で**メインセッションが総合判断**する:

| 基準               | パターン A (並列 agent)            | パターン B (複数 agent)              |
| ------------------ | ---------------------------------- | ------------------------------------ |
| **タスク間の連携** | 各タスクは独立、結果を集約するだけ | タスク間で発見を共有・相互検証が有益 |
| **タスク数**       | 2〜3 個                            | 4 個以上                             |
| **タスクの複雑度** | 単純・短時間で完了                 | 複雑・長時間・複数ファイル変更       |
| **コンテキスト**   | 単一モジュール内                   | 複数モジュール/レイヤーにまたがる    |
| **最適なケース**   | 結果だけが重要な集中タスク         | 議論とコラボが必要な複雑タスク       |

**目安:**

- タスク数 3 以下 かつ 各タスクが単純 → パターン A 推奨
- タスク数 4 以上 → パターン B 推奨
- タスク間で相互検証が有益 → パターン B 推奨
- 同一ファイルへの変更が複数タスクにまたがる → パターン A 推奨 (競合回避)

### 3. タスクの並列実行

#### パターン A: 並列 agent

各タスクを `multi_tool_use.parallel` または複数の `spawn_agent` で**単一ターン内に並列起動**する。

```
spawn_agent({
  description: "<タスクの要約>",
  prompt: `以下のタスクを実行してください:

<タスクの詳細な指示>

完了したら結果をマークダウン形式で報告してください。`
})
```

**重要:**

- `spawn_agent` は、ユーザーが明示的に delegation / sub-agents / parallel agents を求めた場合のみ使う
- `spawn_agent` を使わない場合は、`multi_tool_use.parallel` で独立した read / bash / 調査を並列化する
- 各 agent の結果はメインセッションで収集し、最後に一括して統合する

→ ステップ 4 (結果の集約) へ進む

#### パターン B: 複数 agent

各タスクに specialist agent を割り当て、独立したセッションで真に並列実行する。agent 間の直接連携は前提にせず、メインセッションが結果を集約する。

**詳細な手順は `references/agent-teams-pattern.md` を参照。** 概要:

1. **B-1.** `spawn_agent` をタスク数分だけ単一メッセージ内で並列起動
2. **B-2.** 返ってきた agent id を保持する
3. **B-3.** `wait_agent` で完了を待ち、結果を回収する
4. **B-4.** 不要になった agent は `close_agent` で閉じる

**フォールバック:** `spawn_agent` 失敗 → パターン A / 全 agent 失敗 → パターン A / 一部失敗 → 残りの結果で続行

### 4. 結果の集約と報告

全タスクの結果を集約してユーザーに報告する。**報告テンプレートは `references/report-format.md` を参照。**

報告に含める情報:

- 実行方式 (並列 agent / 複数 agent)
- タスク数と成功・失敗の内訳
- 各タスクの結果要約
- 変更ファイル一覧 (あれば)
- 失敗したタスクのエラー内容と推奨対応

## 依存関係のあるタスクの処理

タスク間に依存関係がある場合:

1. 依存関係のないタスクを先に並列実行
2. 依存先タスクの完了を待機
3. 依存タスクを実行 (必要なら再度並列)

## エラーハンドリング

### タスクの一部が失敗した場合

成功したタスクの結果は保持し、失敗したタスクのみを報告する。ユーザーに再実行するか確認:

```
失敗したタスクだけを再実行するか、成功分のみで完了するかを短く確認する。
```

### 全タスクが失敗した場合

- パターン B (複数 agent) で全 agent が失敗 → パターン A (並列 agent) にフォールバックして再試行
- パターン A で全 agent が失敗 → エラー内容を報告し、ユーザーに個別実行を提案

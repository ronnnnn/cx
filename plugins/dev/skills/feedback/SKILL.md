# フィードバック検証・反映ワークフロー

引数で受け取ったフィードバックやレビュー内容を検証し、現在のコンテキストに照らして「修正すべき指摘」だけを反映する。妥当な指摘でもスコープ外・方針と異なる場合は修正せず、理由付きで報告する。

## 重要な原則

1. **妥当性を必ず検証する** - 指摘を鵜呑みにせず、実際のコード、差分、ドキュメントを確認し、事実誤認・論理矛盾・既対応がないか吟味する
2. **現在のコンテキストを基準にする** - 修正の要否は、セッションの方針、ユーザーの明示指示、ブランチ差分に照らして判断する
3. **スコープ外・方針違いは修正しない** - 妥当な指摘でも今回の変更範囲外、または確立済みの方針と異なるなら反映せず報告のみ行う
4. **判断が不明瞭なら確認する** - 妥当性、スコープ、方針の判断がつかない場合は、推測せずユーザーに短く確認する
5. **修正対象は即実施する** - 修正対象と判定した指摘は、既存の設計・規約・スタイルに合わせて反映し、検証まで行う

## 現在のコンテキストの定義

「現在のコンテキスト」は次の 2 要素で構成する。

1. **セッション中のコンテキスト情報** - これまでの会話で確立した目的、設計判断、方針、制約、既知の検討事項
2. **差分** - 現在のブランチに応じて取得方法を切り替える。デフォルトブランチは `origin/HEAD` から検出し、`main` は決め打ちしない

差分の取得方針:

- **デフォルトブランチ以外**: 分岐元 (merge-base) から現在のブランチまでの差分
- **デフォルトブランチ**: staged + unstaged の差分

## 作業開始前の準備

指摘が 3 件以上、または検証に複数ファイルの確認が必要な場合は `update_plan` でステップを管理する。1〜2 件の単純な検証なら、タスク登録なしで直接進めてよい。

タスク登録する場合は、現在の作業に合わせて `update_plan` 全体を更新し、不要な古いステップを残さない。

```
update_plan({
  plan: [
    { step: "フィードバックを取得して指摘単位に分割", status: "pending" },
    { step: "現在のコンテキストと差分を収集", status: "pending" },
    { step: "各指摘の妥当性・スコープ・方針整合性を判定", status: "pending" },
    { step: "修正対象の指摘を反映", status: "pending" },
    { step: "検証して判定結果を報告", status: "pending" }
  ]
})
```

各ステップの開始時に `update_plan` で `in_progress` にし、完了時に `completed` に更新する。

## 実行手順

### 1. フィードバックの取得と分割

引数があれば、それを検証対象とする。複数の指摘が含まれる場合は、独立して判定できる個別の指摘単位に分割する。

分割時は次を認識する:

- 番号付きリスト
- 箇条書き
- レビューコメント単位
- 段落単位
- 「と」「、」で接続された独立指摘

引数がなく、直近の会話から対象が明確でない場合は、検証・反映するフィードバックやレビュー内容をユーザーに短く確認してから進める。直近のコードやログから勝手に対象を決めない。

### 2. 現在のコンテキストの収集

#### 2-1. セッション方針の把握

これまでの会話から、今回の作業の目的、スコープ、確立済みの設計判断、意図的な選択を整理する。これが後段の「方針整合性」判定の基準になる。

#### 2-2. ブランチ差分の取得

現在のブランチを判定し、差分を取得する。

```bash
if ! git rev-parse --is-inside-work-tree >/dev/null 2>&1; then
  echo "Git リポジトリ外: ブランチ差分なし (セッションのコンテキストで検証)"
else
  current=$(git rev-parse --abbrev-ref HEAD)
  default=$(git symbolic-ref --quiet --short refs/remotes/origin/HEAD 2>/dev/null | sed 's@^origin/@@')

  if [ "$current" = "$default" ]; then
    git diff HEAD
  else
    base=""
    if [ -n "$default" ]; then
      base=$(git merge-base "$default" HEAD 2>/dev/null || git merge-base "origin/$default" HEAD 2>/dev/null)
    fi

    if [ -n "$base" ]; then
      git diff "$base"
    else
      echo "分岐元を特定できないためブランチ差分なし (セッションのコンテキストで検証)"
    fi
  fi
fi
```

`git diff "$base"` は分岐点から作業ツリーまでの全変更 (コミット済み + 未コミット) を含む。指摘がブランチでの作業全体に対するものか、未コミット分のみかを区別したい場合は `git diff "$base" HEAD` も併用する。

差分が空の場合でも、セッションのコンテキストを基準に検証を続ける。

### 3. 各指摘の検証と判定

各指摘について、関連ファイルを読み込んだ上で次の 3 観点を順に判定する。詳細な基準は `references/evaluation-criteria.md` を参照する。

1. **妥当性 (Validity)** - 指摘は技術的に正しいか。事実誤認・論理矛盾はないか。既に対応済みでないか
2. **スコープ (Scope)** - 今回のコンテキスト (差分とセッションの目的) が扱う範囲内か
3. **方針整合性 (Alignment)** - セッションで確立した方針、意図的な選択、プロジェクト規約と一致するか

判定結果で各指摘を分類する。

| 妥当性 | スコープ | 方針 | 分類                  |
| ------ | -------- | ---- | --------------------- |
| OK     | 内       | 一致 | 修正対象              |
| OK     | 外       | -    | 報告のみ (スコープ外) |
| OK     | 内       | 違い | 報告のみ (方針違い)   |
| NG     | -        | -    | 報告のみ (妥当性なし) |
| 不明   | -        | -    | 確認                  |

#### ファクトチェック

技術的な主張を含む指摘は、最新ドキュメントと、プロジェクトが実際に使用している言語・ツールの API バージョンに沿った情報で裏付けを取る。特に次のケースでは必ず実施する。

- 言語仕様、ランタイムの挙動、フレームワーク・ライブラリの API や推奨パターン
- セキュリティ・パフォーマンスに関する主張
- 「すべき」「非推奨」など規範的・断定的な主張

手順:

1. **使用バージョンを特定する** - 依存マニフェスト、ロックファイル、ランタイム指定ファイルを確認する
2. **そのバージョンの情報で照合する** - 指摘が前提とする API、挙動、推奨が、対象バージョンに実在し有効か確認する
3. **最新動向も確認する** - 非推奨、破壊的変更、新 API の有無を最新ドキュメントで確認する

ソースの優先順位:

| 優先度 | ソース       | 用途                                   |
| ------ | ------------ | -------------------------------------- |
| 1      | LSP / repo   | コードベース内の定義・参照・型情報     |
| 2      | deepwiki MCP | OSS リポジトリの Wiki・ドキュメント    |
| 3      | Gemini MCP   | Google 検索による最新情報              |
| 4      | context7 MCP | ライブラリの公式ドキュメントとコード例 |
| 5      | WebSearch    | 最新情報・公式サイト・GitHub           |

MCP や追加 tool がまだ有効化されていない場合は、`tool_search` で必要な tool を探してから使う。OpenAI、Terraform、Google Cloud など専用 MCP がある領域は、それぞれの専用 tool / skill を優先する。

### 4. 不明瞭な場合の確認

妥当性、スコープ、方針のいずれかが判断できない指摘は、推測で処理せずユーザーに確認する。指摘ごとに細かく質問するのではなく、判断がつかない箇所をまとめて 1 回で確認する。

確認が必要な場合は、何が分かれば判定できるのかを明示する。

### 5. 修正の実施

「修正対象」と分類した指摘を即座に反映する。

- 既存コードのスタイル・規約・命名に合わせる
- 1 つの指摘に複数箇所の変更が必要な場合は漏れなく反映する
- 修正によって新たな不整合、型エラー、未使用、参照切れが生じないか確認する
- プロジェクトのタスクランナーがあれば、整形・検証を実行する

### 6. 完了報告

判定結果と対応内容をまとめて報告する。報告フォーマットは `references/evaluation-criteria.md` の「報告フォーマット」を参照する。

最低限、次を含める:

- **修正を実施した指摘**: 指摘内容、対応内容、変更ファイル
- **修正を見送った指摘**: 指摘内容、判定、理由
- **確認が必要な指摘**: 判断できない理由、必要な情報
- **検証結果**: 実行した format、test、lint、型チェック、spot check

## エラーハンドリング

### Git リポジトリ外で実行した場合

ブランチ差分は取得できない旨を伝え、セッションのコンテキスト情報のみを基準に検証を続ける。差分による裏付けができない判定は、その旨を報告に明記する。

### 差分が空の場合

ブランチ差分が取得できなくても、セッションで確立した方針を基準に妥当性・方針整合性を判定する。スコープ判定は差分がない前提で慎重に行う。

### 指摘がすべて修正不要だった場合

修正は行わず、各指摘について見送り理由を明記して報告する。妥当な指摘を「スコープ外」「方針違い」として見送る場合は、後続で対応する余地があることも併記する。

### 関連ファイルが読み込めない場合

該当指摘の妥当性検証をスキップし、確認が必要な旨を報告に明記する。

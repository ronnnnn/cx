# 判定基準と報告フォーマット

`feedback` スキルで各指摘を検証・分類・報告する際の詳細基準。

## 判定の 3 観点

各指摘を、関連コードと差分を実際に確認した上で次の順に判定する。1 つでも下流の前提を満たさない場合は、それ以降の観点を評価するまでもなく分類が確定する。

### 1. 妥当性 (Validity)

指摘が技術的・論理的に正しいかを吟味する。指摘を鵜呑みにせず、客観的な根拠で判断する。技術的な主張は、最新ドキュメントとプロジェクトが実際に使う API バージョンに沿ってファクトチェックする。

妥当と判断する観点:

- 事実関係が正しい
- 指摘の理由付けに論理的な矛盾がない
- 言語・フレームワークの慣習やベストプラクティスに照らして正当
- 指摘されたバグ・リスクが実際に再現・成立しうる
- セキュリティ・パフォーマンス・可読性・保守性の観点で実益がある
- プロジェクトが使うバージョンに合致している

妥当でないと判断する観点:

- 指摘がコードを誤読している
- すでに差分内または既存コードで対応済み
- 論理が破綻している、または根拠が示されていない
- 古い情報・非推奨パターンに基づく指摘
- プロジェクトが使うバージョンに存在しない API や挙動を前提にしている
- トレードオフを無視した一面的な指摘で、現状の選択の方が合理的

### 2. スコープ (Scope)

指摘が今回のコンテキスト (差分とセッションの目的) が扱う範囲内かを判断する。

スコープ内と判断する観点:

- 今回の差分が追加・変更した箇所に対する指摘
- 今回の作業目的を達成するために必要な修正
- 今回の変更が直接引き起こした副作用・不整合への指摘

スコープ外と判断する観点:

- 差分と無関係な既存コードへの指摘
- 今回の作業目的に含まれない別タスク・別 PR で扱うべき改善
- リファクタリング・大規模な設計変更など、今回の変更粒度を超えるもの
- ついでに直せる程度でも、本来の目的から逸脱するもの

### 3. 方針整合性 (Alignment)

指摘が、セッションで確立した方針や意図的な選択と一致するかを判断する。

方針一致と判断する観点:

- セッションで合意・確立した設計判断と整合する
- プロジェクトの規約 (`AGENTS.md` など) に沿う
- 既存の意図的な選択と矛盾しない

方針違いと判断する観点:

- セッションで意図的に選択した実装・設計と相反する
- プロジェクトの規約・既存パターンと相反する
- 過去に検討して却下した方針への揺り戻し

方針違いの指摘は「妥当だが今回は採用しない」ケースとして扱い、なぜ現在の方針を採るのかを根拠とともに報告する。

## 分類マトリクス

| 妥当性 | スコープ | 方針 | 分類     | アクション               |
| ------ | -------- | ---- | -------- | ------------------------ |
| OK     | 内       | 一致 | 修正対象 | 即修正                   |
| OK     | 外       | -    | 報告のみ | スコープ外として報告     |
| OK     | 内       | 違い | 報告のみ | 方針違いとして報告       |
| NG     | -        | -    | 報告のみ | 妥当性に欠ける根拠を報告 |
| 不明   | -        | -    | 確認     | ユーザーに確認           |

## 報告フォーマット

```markdown
## フィードバック検証結果

検証対象: N 件 (修正 X 件 / 見送り Y 件 / 確認 Z 件)

### 修正を実施 (X 件)

| 指摘                          | 対応内容                   | 変更ファイル       |
| ----------------------------- | -------------------------- | ------------------ |
| エラー時に retry されていない | 失敗時のリトライ処理を追加 | src/api/client.ts  |
| null 安全でない               | オプショナルチェーンを導入 | src/utils/parse.ts |

### 修正を見送り (Y 件)

| 指摘                           | 判定       | 理由                                                 |
| ------------------------------ | ---------- | ---------------------------------------------------- |
| 全体をクラス化すべき           | 方針違い   | 本セッションでは関数ベースの簡潔さを優先するため     |
| 別モジュールの命名も統一すべき | スコープ外 | 今回の差分に含まれない既存コードへの指摘のため       |
| この箇所はメモリリークする     | 妥当性なし | 当該値は関数スコープで解放されるため指摘は成立しない |

### 確認が必要 (Z 件)

| 指摘                 | 判断できない理由               | 必要な情報              |
| -------------------- | ------------------------------ | ----------------------- |
| API を新方式に寄せる | 採用方針が会話から判断できない | 今回の scope に含めるか |

### 検証

- `bun run fmt`
- `bun test`
- `rg '<確認した文字列>'`
```

## 報告時の注意

- 修正したものは、何をなぜ直したかが分かる粒度で簡潔に書く
- 見送ったものは、なぜ今回は直さないかを根拠付きで書く
- スコープ外・方針違いで見送った妥当な指摘は、別タスクで扱える余地があれば補足する
- 実行できなかった検証があれば理由と残リスクを明示する

# Plan

引数として渡された資料 (ファイル、URL、PR、設計書等) を分析し、アトミックな粒度のタスクに分解した行動計画を作成する。

## 作業開始前の準備

**必須:** 作業開始時に `update_plan` で実行ステップを整理し、以下のステップを登録する:

```
update_plan({
  plan: [
    { step: "資料の読み込みと分析", status: "pending" },
    { step: "要件の明確化", status: "pending" },
    { step: "出力先の確認", status: "pending" },
    { step: "アトミックなタスクへの分解", status: "pending" },
    { step: "依存関係の整理とフェーズ構成", status: "pending" },
    { step: "行動計画の出力", status: "pending" },
    { step: "レビューと修正", status: "pending" },
    { step: "完了報告", status: "pending" },
  ]
})
```

各ステップの開始時に `update_plan` で `in_progress` にし、完了時に `completed` に更新する。

## 処理フロー

### Step 1: 資料の読み込みと分析

引数として渡された資料を読み込み、全体像を把握する:

- **ファイルパス**: Read ツールで読み込み
- **URL**: 必要時のみ WebFetch で取得
- **PR URL**: `gh pr view` と `gh pr diff` で概要と差分を取得
- **複数資料**: 全てを読み込み、関連性を整理

資料から以下を抽出する:

- 最終的なゴール (何を達成するか)
- スコープ (対象範囲と制約条件)
- 技術的な前提条件
- 既知のリスクや懸念事項

### Step 2: 要件の明確化

資料を分析した結果、不明点や曖昧な箇所がある場合は ユーザーへの短い確認 でユーザーにインタビューする。不明点がなくなるまで以下のサイクルを繰り返す:

1. 不明点・曖昧な箇所を洗い出す
2. ユーザーへの短い確認 で具体的に質問する
3. 回答を踏まえて理解を更新する
4. まだ不明点があれば 1 に戻る

質問の観点:

- ゴールの優先順位やスコープの境界
- 技術的な制約や前提条件
- 既存のコードやアーキテクチャとの整合性
- 非機能要件 (パフォーマンス、セキュリティ等)
- 判断が分かれる設計上の選択肢

不明点がない場合はこのステップをスキップする。

### Step 3: 出力先の確認

ユーザーへの短い確認 で行動計画の出力先を尋ねる。選択肢の例:

- **`.claude/plans/`**: プロジェクト内の計画専用ディレクトリ
- **`docs/plans/`**: ドキュメントディレクトリ配下
- **カスタムパス**: ユーザー指定のパス

出力先ディレクトリが存在しない場合は作成する。

### Step 4: アトミックなタスクへの分解

資料の内容をこれ以上分割できない最小かつアトミックな粒度のタスクに分解する。

#### 分解の基準

- **単一の成果物**: 1 タスクで 1 つの具体的な成果物を生む
- **明確な開始と終了**: 何をすれば完了か曖昧でない
- **独立した検証**: タスク単体で動作確認やテストが可能
- **推定作業量**: Codex の 1 ターンで完了可能な粒度

#### 実行主体の分類

各タスクを以下のいずれかに分類する:

- **Codex** (🤖): コード生成、ファイル編集、テスト実行、Git 操作、検索・分析等
- **ユーザー手動** (👤): 外部サービス設定、UI 操作確認、環境構築、アカウント作成、デプロイ承認等
- **Codex + ユーザー確認** (🤝): Codex が実行し、ユーザーが結果を確認

手動タスクには必ず具体的な手順を記載する。

#### 完了条件の設定

各タスクの最後に検証可能な完了条件を必ず記載する。自動検証 (テスト、Lint、ビルド、型チェック) を優先し、自動化が困難な場合のみ手動検証とする。手動検証の場合は確認手順と期待結果を詳細に記載する。

具体的な記述パターンは `references/plan-format.md` の「完了条件の記述パターン」を参照。

### Step 5: 依存関係の整理とフェーズ構成

#### 依存関係の整理

タスク間の依存関係を明らかにする:

- **逐次実行**: 先行タスクの完了が前提条件 → `depends_on` で明示
- **並列実行可能**: 相互に独立 → 同一フェーズ内で並列実行可能と明記
- **部分依存**: 先行タスクの一部の成果物のみ必要 → 依存の粒度を明記

依存関係はタスク ID (例: `T1`, `T2`) で参照する。

#### フェーズ構成

関連するタスクをフェーズにまとめる:

- 依存関係が近いタスクを同一フェーズに配置
- フェーズ間の依存関係を明確にする
- 各フェーズに目的と概要を付与

### Step 6: 行動計画の出力

フェーズごとにファイルを分割して出力する。

#### ファイル構成

```
<出力先>/
├── 00-overview.md        # 全体概要、フェーズ一覧、前提条件
├── 01-phase-name.md      # フェーズ 1 のタスク一覧
├── 02-phase-name.md      # フェーズ 2 のタスク一覧
├── ...
└── NN-phase-name.md      # フェーズ N のタスク一覧
```

- フェーズが大きい場合はさらにタスク単位でファイルを分割可能
- ファイル名はゼロパディングの連番 + ケバブケースのフェーズ名

#### 記述フォーマット

各ファイルのフォーマットは `references/plan-format.md` を参照。

### Step 7: レビューと修正

行動計画を出力後、ユーザーにレビューを依頼する。フィードバックがある場合は修正サイクルを繰り返す:

1. ユーザーに行動計画のレビューを依頼する
2. フィードバック (不明点、修正要望、追加要件) を受け取る
3. フィードバックに基づき該当ファイルを修正する
4. 修正内容をユーザーに報告する
5. さらにフィードバックがあれば 2 に戻る

ユーザーが承認した場合、次のステップに進む。

### Step 8: 完了報告

全ファイル出力後、以下を報告する:

- 作成したファイルの一覧
- 総タスク数 (Codex / 手動 / 確認の内訳)
- クリティカルパス (最長の依存チェーン)
- 並列実行可能なタスクグループ
